Section 3

Inheritance Hierarchy

The metamodel is organized into a three-level inheritance tree rooted at the abstract GUIElement class. This hierarchy cleanly separates containers (which hold children) from widgets (leaf elements).

GUIElement (name: String) ← abstract root
├── Window (title, width, height, layout, children) ← top-level container
├── Panel (title, layout, borderTitle, children) ← nested container
└── Widget (position) ← abstract leaf base
    ├── Label (text: String)
    ├── Button (label: String)
    ├── TextField (columns: Integer, text: String)
    ├── TextArea (rows, columns: Integer, scrollable: Boolean)
    ├── CheckBox (label: String, selected: Boolean)
    ├── ComboBox (items: List<String>, selectedIndex: Integer)
    ├── Slider (min, max, value: Integer)
    └── Separator (orientation)

Supporting types (internal):
├── Layout (type, rows, cols, hgap, vgap)
├── LayoutType {HORIZONTAL, VERTICAL, FLOW, GRID, BORDER}
└── Position {TOP, BOTTOM, LEFT, RIGHT, CENTER}
Key Insight: Window and Panel are both containers that hold children, but they are modeled as sibling subclasses of GUIElement rather than sharing a common "Container" superclass. This is because Window is always the root (with unique properties like size) while Panel is always nested, so merging them would add unnecessary complexity.
Section 4

Class-by-Class Breakdown

4.1 — Abstract Classes

A

GUIElement «abstract»

The root of the entire metamodel hierarchy. Every node in a GUI model — whether a container or a leaf widget — inherits from GUIElement. It provides the universal name attribute (an identifier for referencing) and declares the abstract accept(GUIVisitor) method for the Visitor pattern.

A

Widget «abstract»

The abstract base for all leaf elements — UI controls that do not contain other elements. Adds an optional position attribute (a Position enum value) for layout placement within border-layout containers. All 8 concrete widget types extend this class.

4.2 — Container Classes

W

Window

The top-level container and the root of every GUI model tree. Has a title, width/height dimensions, a Layout configuration, and an ordered list of GUIElement children. Every QuickGUI program produces exactly one Window. It is always the entry point for interpretation and code generation.

P

Panel

A nested container used for grouping related widgets. Has its own Layout, an optional borderTitle (rendered as a titled border in Swing), and an ordered list of GUIElement children. Panels can nest inside other panels to create arbitrarily deep hierarchies.

4.3 — Concrete Widget Classes

L

Label

A static text display widget. Stores a text string to show to the user. Read-only; no user interaction.

B

Button

A clickable widget. Stores a label string displayed on the button face. Fires an action when clicked.

TF

TextField

A single-line text input. Configurable columns width and optional default text.

TA

TextArea

A multi-line text input. Has rows, columns dimensions, and a scrollable flag.

CB

CheckBox

A boolean toggle control. Stores a label and a selected boolean state.

CX

ComboBox

A dropdown selection widget. Holds a list of items (strings) and a selectedIndex.

S

Slider

A range selection widget. Defined by min, max, and current value integers.

—

Separator

A visual divider line (horizontal or vertical). Has no interactive behavior — purely structural.

4.4 — Supporting Configuration Class

⚙

Layout

A configuration object that specifies how a container arranges its children. Contains a LayoutType enum, optional rows/cols for grid layouts, and hgap/vgap spacing. Not exposed in the DSL — the framework assigns layouts automatically (defaulting to VERTICAL stacking).

Section 6

Multiplicity Analysis

The multiplicities on relationships define the structural constraints of valid GUI models. They express the cardinality rules that every well-formed model must satisfy.

Container → Children (Composition)
Window
GUIElement

Exactly one container owns zero or more children. Children cannot exist without a parent.

Container → Layout (Composition)
Window / Panel
Layout

Every container has exactly one layout. The layout cannot be shared or orphaned.

Layout → LayoutType (Association)
Layout
LayoutType

Each layout references exactly one layout type enum value (VERTICAL, HORIZONTAL, etc.).

Composition vs. Association: The filled diamond (♦) denotes composition — the child's lifecycle is bound to the parent. When a Window is destroyed, its children and layout are destroyed too. The open arrow (→) denotes association — a reference to a shared or independent value (like an enum constant).
Section 7

Enumeration Types

The metamodel uses two enumeration types — both are internal (not exposed to DSL users). They constrain configuration values to a fixed set of valid options.

«enumeration» LayoutType

Defines the strategy a container uses to arrange its children:

HORIZONTAL VERTICAL FLOW GRID BORDER

Maps to Java Swing layout managers: BoxLayout (Y/X), FlowLayout, GridLayout, BorderLayout. Default is VERTICAL (stack top-to-bottom).

«enumeration» Position

Positional constraint for widgets inside a BorderLayout container:

TOP BOTTOM LEFT RIGHT CENTER

Maps directly to BorderLayout.NORTH, SOUTH, WEST, EAST, CENTER. Optional — only meaningful when the parent uses BORDER layout.

Section 8

Design Patterns

The metamodel employs four well-known design patterns, each serving a specific architectural purpose. Together they provide extensibility, type safety, and clean separation of concerns.

Composite Pattern

Window and Panel act as composites that hold an ordered list of GUIElement children. Widgets are leaves. This enables a uniform tree structure where containers and widgets are treated polymorphically, allowing Panels to nest inside other Panels to any depth.

Classes: GUIElement, Window, Panel, Widget (and subclasses)
Visitor Pattern

The GUIVisitor interface declares visit() overloads for each concrete element type. Each GUIElement implements accept(GUIVisitor) to dispatch to the correct visitor method. This decouples what we do with the model from what the model is — new semantics can be added without modifying any model class.

Classes: GUIVisitor, SwingInterpreter, JavaCodeGenerator, all GUIElements
Builder Pattern

WindowBuilder and PanelBuilder implement the Builder pattern to construct complex model objects step-by-step through a fluent method-chaining API. The builders accumulate configuration calls and produce the final Window/Panel via terminal operations (.build(), .show()).

Classes: GUI, WindowBuilder, PanelBuilder, ContainerBuilder
CRTP (Curiously Recurring Template Pattern)

ContainerBuilder<SELF> uses a type parameter bound to the concrete builder type. This ensures that fluent methods like .label() return the correct builder type (WindowBuilder or PanelBuilder) rather than the abstract base — enabling type-safe chaining without casts.

Classes: ContainerBuilder<SELF>, WindowBuilder, PanelBuilder
Section 9

DSL Builder Layer

The builder classes form the concrete syntax of the DSL — they translate fluent Java method chains into metamodel instances. The diagram shows the builder layer as a separate concern connected to the metamodel via dependency (dashed arrow) and builds relationships.

Builder Architecture

Builder class hierarchy 
// Abstract base — shared widget-adding API via CRTP
public abstract class ContainerBuilder<SELF extends ContainerBuilder<SELF>> {
    protected abstract SELF self();
    protected abstract void addChildToModel(GUIElement child);

    public SELF label(String text)    { /* add Label model */ }
    public SELF button(String text)   { /* add Button model */ }
    public SELF textField(...)        { /* add TextField model */ }
    // ... all widget methods return SELF for chaining

    public PanelBuilder<SELF> panel(String title) {
        return new PanelBuilder<>(title, self(), this::addChildToModel);
    }
}

// Concrete — builds Window models
public class WindowBuilder extends ContainerBuilder<WindowBuilder> {
    public WindowBuilder size(int w, int h) { ... }
    public Window        build()              { ... }  // terminal
    public Window        show()               { ... }  // terminal
    public String        generateCode(...)    { ... }  // terminal
}

How the DSL Constructs Models

Each fluent method call in the DSL creates a metamodel object and adds it to the current container:

  1. GUI.window("Login") — Creates a new WindowBuilder wrapping a fresh Window model object
  2. .label("text") — Creates a Label model and adds it to the window's children list
  3. .panel("Form") — Creates a PanelBuilder wrapping a new Panel; switches the active builder context
  4. .endPanel() — Adds the completed Panel to the parent's children and returns the parent builder
  5. .show() — Terminal operation: triggers the SwingInterpreter visitor to render the model as a live GUI

DSL Usage Example

LoginFormExample.java 
GUI.window("Login")
   .size(350, 250)
   .label("Please sign in")
   .panel("Form")                // → PanelBuilder (context switch)
       .label("Username:")
       .textField("username")
       .label("Password:")
       .textField("password")
       .checkBox("Remember me")
   .endPanel()                   // → WindowBuilder (back to parent)
   .button("Login")
   .button("Cancel")
   .show();                      // terminal → live Swing GUI
DSL Design Constraint: Every DSL method accepts only strings and numbers — no Java-specific types, enums, lambdas, or layout managers appear in the API surface. Layout and positioning are handled automatically by the framework, keeping the DSL minimal and accessible.
Section 10

Semantics Layer (Visitor-Based)

The Visitor pattern completely decouples model structure from model meaning. Two concrete visitors provide independent semantics for the same metamodel:

▶

SwingInterpreter

Walks the model tree and interprets it by creating live Java Swing components. Each visit() method maps a model element to its corresponding Swing widget (e.g., Window → JFrame, Button → JButton). The visitor recursively processes container children to build the complete UI.

⟨/⟩

JavaCodeGenerator

Walks the model tree and generates standalone, compilable Java source code. Each visit() method emits the Swing API calls needed to reproduce the GUI. The output is a self-contained .java file that can be compiled and run independently.

Open for Extension: Adding a new semantics (e.g., an HTML renderer, a FXML exporter, or a validation visitor) requires only implementing the GUIVisitor interface — no modifications to any model class are needed. This is the core benefit of the Visitor pattern in this architecture.

Summary

Key Takeaways

🏗️ Three-Layer Architecture

The metamodel (abstract syntax), DSL builders (concrete syntax), and visitors (semantics) form cleanly separated layers. Each can evolve independently.

🌳 Composite Tree Structure

GUI models form a tree: Window at the root, Panels as interior nodes, and Widgets as leaves. The Composite pattern enables uniform traversal and arbitrary nesting depth.

🔌 Open Extension via Visitor

New semantics (interpretation, code generation, validation, export) can be added by implementing the GUIVisitor interface — no model modifications needed.

🔗 Type-Safe Fluent API

The CRTP-based builder hierarchy ensures that method chains are type-checked at compile time. You cannot call .show() inside a panel or .endPanel() at the window level.

📐 Strings & Numbers Only

The DSL surface accepts only primitive types — no Java-specific enums, lambdas, or layout managers. Implementation complexity is hidden behind the fluent API.

⚙️ Automatic Layout

Layout is an internal concern, not exposed to DSL users. Containers default to VERTICAL stacking, making the DSL simple while the framework handles the visual arrangement.
